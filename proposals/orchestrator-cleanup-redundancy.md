## 중복 1: 기어 판단 기준 — 코드 블록 vs 테이블 이중 표현

**위치**: 줄 49-84 (코드 블록 ASCII 다이어그램) vs 줄 87-92 (테이블)

**겹치는 내용**: 기어 1-4의 판단 기준(파일 수, 변경 규모)과 각 기어에 해당하는 Phase가 두 가지 형식으로 거의 동일하게 반복된다.

- 코드 블록: "1개 파일, 단순 수정 → 기어 1" / "3-10 파일, 신규 기능 → 기어 3"
- 테이블: "1줄 수정, 오타, 로그 추가" / "3-10 파일, 신규 기능"

**통합 제안**: 코드 블록(줄 49-84)에서 기어별 판단 기준 부분을 제거하고, **판별 흐름(Yes/No 분기)**만 남긴다. 판단 기준의 정확한 정보는 테이블(줄 87-92)로 일원화. 코드 블록은 각 기어의 "흐름(Phase 순서)"만 보여주는 용도로 축소.

```
변경 전 (코드 블록):
  기어 1 (즉시):
    state.json 생성 안 함...
    사용자: "이 오타 수정해줘"
    흐름: 바로 구현 → lint → 커밋
    마찰: 제로
  ... (기어 2, 3, 4 모두 유사 반복)

변경 후 (코드 블록):
  판별 기준 다이어그램만 유지 (줄 75-84)
  각 기어의 흐름은 테이블로 통합
```

## 중복 2: 기어별 산출물 테이블과 기어별 분기 코드 블록의 state.json 정보 중복

**위치**: 줄 49-70 (기어별 분기 코드 블록, 특히 state.json 관련) vs 줄 96-104 (기어별 산출물 테이블)

**겹치는 내용**: state.json 생성 여부가 양쪽 모두에 기술됨.
- 코드 블록: "기어 1: state.json 생성 안 함" / "기어 2: state.json = 미니멀"
- 테이블: "기어 1: state.json 생성 X" / "기어 2: 미니멀"

**통합 제안**: 중복 1의 통합과 연계하여, 기어별 분기 코드 블록에서 state.json 관련 내용을 제거. 산출물 테이블(줄 96-104)에 state.json 정보가 이미 포함되어 있으므로 충분.

## 중복 3: Phase status 흐름 — 섹션 3-2 vs 섹션 4-2 vs 섹션 5

**위치**:
- 줄 136-147 (섹션 3-2, `/workflow next` 2단계 동작 코드 블록)
- 줄 375-383 (섹션 4-2, Phase status 값 정의 테이블)
- 줄 462-473 (섹션 5, 상태 전이도의 Phase 내부 루프)

**겹치는 내용**: `in_progress → reviewing → needs_revision → approved` 흐름이 세 곳에서 각각 다른 형식으로 반복.

- 섹션 3-2: 코드 블록으로 1번째/2번째 next 동작 설명
- 섹션 4-2: 테이블로 각 status의 의미와 전환 트리거
- 섹션 5: ASCII 다이어그램으로 Phase 내부 루프

**통합 제안**:
- 섹션 3-2(줄 136-147)를 **동작 관점의 정의(canonical)**로 유지 (사용자가 커맨드를 어떻게 사용하는지)
- 섹션 4-2(줄 375-383)를 **status 값의 기술 레퍼런스**로 유지 (스키마 문서 성격)
- 섹션 5(줄 462-473)의 Phase 내부 루프 부분은 "상세는 섹션 3-2 참조"로 축약. 현재 전이도의 핵심인 Phase 간 이동만 남기고, 내부 루프 다이어그램은 제거.

## 중복 4: designStale — 섹션 3-3 vs 섹션 4-4

**위치**: 줄 192-204 (섹션 3-3, `/workflow back` 장거리 에스컬레이션 설명) vs 줄 394-406 (섹션 4-4, `artifacts.designStale` 전용 섹션)

**겹치는 내용**: designStale의 설정 조건과 동작이 양쪽에서 거의 동일하게 설명됨.

- 섹션 3-3: "Implement에서 `back specify` 시 → Design 문서에 `stale: true` 표시 → Design 재진입 시 갱신 필요 안내"
- 섹션 4-4: "Implement 또는 Verify에서 `back specify` 에스컬레이션 시 자동으로 `true` 설정 → Design Phase 재진입 시 갱신 필요 안내 → 완료 후 `false` 복원"

**통합 제안**: 섹션 3-3에서는 designStale 동작의 상세 설명을 제거하고 "중간 Phase의 문서는 stale 표시됨 (상세: 섹션 4-4)"으로 참조만 남긴다. designStale의 정의와 동작은 섹션 4-4에 일원화.

## 중복 5: Slice 재작업 — 섹션 3-3 vs 섹션 8-3 진입 경로

**위치**: 줄 206-220 (섹션 3-3, Slice 단위 부분 재작업) vs 줄 860-877 (섹션 8-3, Implement Phase의 status별 진입 경로 + 프로세스)

**겹치는 내용**: `--slice N`으로 특정 슬라이스만 재작업하고 나머지는 유지한다는 설명이 반복.

- 섹션 3-3: 커맨드 사용법 + 출력 예시로 설명
- 섹션 8-3: Phase 진입 경로 + 프로세스 단계에서 동일 동작 재설명

**통합 제안**: 이 중복은 **의도적 중복**(커맨드 레퍼런스 vs Phase 지침)에 해당하므로 유지해도 무방. 다만 섹션 8-3의 status별 진입 경로 설명(줄 860-863)을 "상세 커맨드: 섹션 3-3 참조"로 축약하면 간결해진다.

## 중복 6: 세션 복원 흐름 — context-inject.sh vs 섹션 11

**위치**: 줄 582-608 (섹션 6-2, `context-inject.sh` Hook) vs 줄 1199-1236 (섹션 11, 세션 연속성)

**겹치는 내용**: state.json에서 feature, phase, status, loadOnResume 파일을 읽어 컨텍스트를 제공하는 흐름이 양쪽에서 중복.

- context-inject.sh: 코드로 구현된 자동 주입 (UserPromptSubmit)
- 섹션 11: `/workflow resume`의 복원 흐름 5단계

**통합 제안**: 이 중복은 **역할이 다름**(자동 vs 수동)을 구분하지만, context-inject.sh의 출력 예시(줄 601-607)와 섹션 11의 출력 예시(줄 1219-1236)가 유사한 정보를 보여준다. 섹션 11에 "자동 컨텍스트 주입은 `context-inject.sh` Hook이 담당 (섹션 6-2), `/workflow resume`은 추가로 슬라이스 진행률과 다음 작업을 안내"라고 관계를 명시하면 충분.

## 중복 7: 점진적 도입 로드맵의 내용이 본문과 겹침

**위치**: 줄 1240-1256 (섹션 12, 점진적 도입 로드맵) vs 문서 전체 각 섹션

**겹치는 내용**: 로드맵의 각 단계 설명이 이미 본문에서 상세히 다뤄진 내용의 요약 반복.

- "2단계: `phase-guard.sh` Hook 추가" → 섹션 6-2에서 이미 상세 설명
- "4단계: `/workflow next` 2단계 동작" → 섹션 3-2에서 이미 상세 설명
- "8단계: `.review.md` 사이드카" → 섹션 9에서 이미 상세 설명

**통합 제안**: 이 중복은 **의도적**(로드맵은 구현 순서 안내 목적). 제거하지 않되, 각 단계에 본문 섹션 번호를 참조로 추가하면 연결성이 높아진다. 예: "2단계: `phase-guard.sh` Hook 추가 (→ 섹션 6-2)"

## 중복 8: 흐름 예시(섹션 10)에서 본문 설명 재반복

**위치**: 줄 1083-1195 (섹션 10, 흐름 예시) vs 섹션 3 (커맨드 체계) + 섹션 8 (Phase별 지침)

**겹치는 내용**: 흐름 예시가 커맨드 동작과 Phase 프로세스를 시나리오로 재설명하면서, 섹션 3과 8의 내용을 사실상 반복.

- 예: 줄 1100-1108에서 `/workflow next`의 reviewing 동작 → 섹션 3-2의 동작 설명과 동일
- 예: 줄 1130-1138에서 `/workflow back specify` 동작 → 섹션 3-3의 장거리 에스컬레이션과 동일

**통합 제안**: 이 중복도 **의도적**(예시는 전체 흐름을 보여주는 교육 목적). 제거하지 않되, 현재 2개 예시(10-1 전체 사이클, 10-2 장거리 에스컬레이션)가 모두 장문이므로, 10-2(줄 1172-1195)는 10-1에 이미 장거리 에스컬레이션이 포함되어 있어 **제거 가능**. 10-1이 이미 `back specify` 시나리오(줄 1128-1138)를 보여주고 있다.
