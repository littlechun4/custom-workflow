# Verify 단계 정의서 (Review)

> 작성: 2026-03-09
> 목적: Verify 단계의 역할, 코드 리뷰 프로세스, 에스컬레이션 규칙 정의
> 기어: 3단 (10+ 파일, 아키텍처급) 기준 최대치 설계
> 문서 보존: 산출물 없음 — 리뷰 결과는 state.json에 기록

---

## 산출물 위치

```
산출물 없음
리뷰 결과 → state.json에 기록
코드 수정 → 경미한 건 직접 수정 + 커밋 (구조적 문제만 back)
```

---

## 1. 역할 정의

**"구현된 코드를 질적으로 리뷰하고, 명세 커버리지를 최종 확인한다."**

- 코드 품질 리뷰: 재사용성, 코드 품질, 런타임 효율, 보안
- AC 커버리지 최종 확인: 모든 R-xxx/C-xxx가 테스트로 추적 가능한가
- 경미한 이슈는 직접 수정, 구조적 문제는 적절한 Phase로 back

**Implement와의 역할 분리:**

| | Implement | Verify |
|--|-----------|--------|
| 초점 | "만들기" — TDD로 코드 생산 | "리뷰" — 코드 품질 + 명세 충족 검증 |
| 검증 | auto-gate (테스트/린트/타입 통과) | 질적 리뷰 (재사용성, 효율, 보안) + AC 커버리지 |
| 인간 관여 | 없음 | 기어 3: 있음 (최종 승인) / 기어 2: 없음 |
| 코드 수정 | 주 활동 | 경미한 수정만 (리팩토링, 네이밍 등) |

## 2. 구체적 활동

| 활동 | 주체 | 설명 |
|------|------|------|
| 코드 품질 리뷰 | AI | 재사용성, 가독성, 효율, 보안 관점에서 코드 검토 |
| AC 커버리지 추적 | AI | Spec의 각 R-xxx/C-xxx가 테스트에서 실제 검증되는지 대조 |
| 경미한 수정 | AI | 리뷰에서 발견한 경미한 이슈 직접 수정 + 커밋 |
| 리뷰 결과 보고 | AI | 수정 내역 + 구조적 이슈 + AC 커버리지 결과 정리 |
| 최종 판단 | 인간 (기어 3) | 리뷰 결과 확인 + 비즈니스 의도 일치 확인 + 승인/에스컬레이션 |

## 3. 리뷰 및 반복 다듬기 (Iterative Refinement)

### 전체 흐름

```
[Implement approved] ──[/workflow next]──→ Verify 진입 (in_progress)
  │
  ├─ 1. AI 코드 리뷰 (5개 리뷰 축)
  ├─ 2. AC 커버리지 추적
  ├─ 3. 경미한 이슈 직접 수정 + 커밋
  ├─ 4. 리뷰 결과 보고
  │
  ──[/workflow next]──→
     ├─ 기어 2: 구조적 이슈 없으면 → approved (인간 리뷰 없음)
     │          구조적 이슈 있으면 → needs_revision → AI 수정 → in_progress
     │
     └─ 기어 3: reviewing (인간 리뷰 대기)
          인간이 리뷰 결과 확인
            → 구조적 문제 발견
              → 슬라이스 문제 → /workflow back --slice N
              → 설계 문제 → /workflow back design
              → 요구사항 문제 → /workflow back specify
            → 추가 수정 요청 → needs_revision → AI 수정 → in_progress (draftCount++)
            → 인간 "승인" → approved
  ──[/workflow next]──→ 다음 Phase (Ship)
```

### 단계별 설명

**1단계: AI 코드 리뷰** (in_progress)

AI가 전체 구현 코드를 다음 5개 축으로 리뷰한다 (§5 상세):

1. **재사용성** — 중복 코드, 추출 가능한 패턴, 기존 유틸리티 활용
2. **코드 품질** — 가독성, 네이밍, 구조, 에러 처리
3. **런타임 효율** — N+1, 불필요한 할당, 알고리즘 복잡도
4. **보안** — OWASP Top 10, 입력 검증, 인증/인가
5. **AC 커버리지** — R-xxx/C-xxx → 테스트 매핑 최종 확인

**2단계: 경미한 수정** (in_progress)

리뷰에서 발견한 이슈 중 **경미한 것**은 AI가 직접 수정:
- 네이밍 개선
- 중복 코드 추출
- 불필요한 할당 제거
- 에러 처리 보강
- 수정 후 자동 검증 (lint + type + test) + 커밋

**3단계: 리뷰 결과 보고** (in_progress → `/workflow next`)

AI가 다음을 정리하여 인간에게 보고:
- 수정한 내역 (경미한 수정 목록)
- 구조적 이슈 (back이 필요한 것)
- AC 커버리지 결과

**4단계: 승인** (`/workflow next` 후)

기어별로 동작이 다르다:

- **기어 2**: AI 리뷰에서 구조적 이슈가 없으면 자동 `approved`. 인간 리뷰 없음.
- **기어 3**: `reviewing` (인간 리뷰 대기) 진입.
  - 인간이 리뷰 결과를 확인
  - 코드를 직접 읽고 비즈니스 의도와 일치하는지 판단
  - 추가 수정 요청 → `needs_revision` → AI 수정 → `in_progress` (draftCount++)
  - 구조적 문제 → 에스컬레이션 (§7)
  - 인간 "승인" → `approved`

**승인 권한**: 기어 3에서 `approved`는 인간만 줄 수 있다. 기어 2에서는 AI 리뷰 통과 시 자동 승인.

### 인간 리뷰 방식 (기어 3 전용)

Verify의 인간 리뷰는 **코드에 대한 최종 리뷰**다:

1. **대화형 피드백**: 인간이 코드를 보고 수정 사항을 전달
2. **코드 내 TODO**: `// TODO(review): {코멘트}` 형태로 코드에 직접 마킹
3. **슬라이스 지정**: "Slice B-2의 에러 처리를 보강해라" 식으로 슬라이스 단위 피드백

AI는 인간의 피드백을 받으면:
1. 해당 코드를 수정
2. 자동 검증 (lint + type + test) 재실행
3. 수정 커밋 생성
4. `draftCount++` + `status: in_progress`

### draftCount 상한 (기어 3)

기어 2에서는 인간 리뷰가 없으므로 draftCount가 증가하지 않는다.

```
draftCount ≤ 2: 정상 범위
draftCount = 3: 경고 — "3회째 리뷰 수정입니다. 근본 원인을 먼저 파악하세요."
draftCount ≥ 4: 인터럽트 — 다음 중 선택 필요:
  A) /workflow next --force   (현재 상태로 강제 진행, feedback에 기록)
  B) /workflow back design    (설계 전면 재검토)
  C) /workflow back specify   (요구사항 전면 재검토)
```

### status 전이 요약

> `reviewing`은 Verify에서 **인간 리뷰 대기** 상태를 의미한다. orchestrator §4-2의 범용 정의("AI 자체 검토 중")와 달리, Verify에서는 AI 리뷰가 `in_progress` 중에 완료되고 `/workflow next`로 인간에게 넘기는 시점에 `reviewing`으로 전이한다.

**기어 2:**

| 현재 status | 트리거 | 다음 status |
|-------------|--------|------------|
| `in_progress` | AI 리뷰 + 경미한 수정 완료 + `/workflow next` | `approved` (구조적 이슈 없음) 또는 `needs_revision` (이슈 있음) |
| `needs_revision` | AI 수정 완료 | `in_progress` (draftCount++) |
| `approved` | `/workflow next` | 다음 Phase 전환 (Ship) |

**기어 3:**

| 현재 status | 트리거 | 다음 status |
|-------------|--------|------------|
| `in_progress` | AI 리뷰 + 경미한 수정 완료 + `/workflow next` | `reviewing` (인간 리뷰 대기) |
| `reviewing` | 인간 추가 수정 요청 | `needs_revision` |
| `reviewing` | 인간 에스컬레이션 | back → 적절한 Phase |
| `reviewing` | 인간 "승인" | `approved` |
| `needs_revision` | AI 수정 완료 | `in_progress` (draftCount++) |
| `approved` | `/workflow next` | 다음 Phase 전환 (Ship) |

## 4. 금지사항

| 금지 | 이유 | 대처 |
|------|------|------|
| 구조적 코드 변경 (새 파일, 새 접근법) | Design/Implement 영역 | `/workflow back design` 또는 `/workflow back implement` |
| 요구사항 변경 | Specify 영역 | `/workflow back specify` |
| 설계 변경 | Design 영역 | `/workflow back design` |
| 테스트 새로 추가 | Implement 영역 | `/workflow back --slice N` |
| 문제 무시 | 품질 게이트 무력화 | 반드시 보고 |

### 경미한 수정 vs 구조적 문제 기준

| 경미한 수정 (Verify에서 직접) | 구조적 문제 (back 필요) |
|-------------------------------|------------------------|
| 변수/함수명 개선 | 새 파일/모듈 추가 |
| 중복 코드 → 헬퍼 추출 (같은 파일 내) | 접근법 변경 |
| 불필요한 할당/연산 제거 | 아키텍처 변경 |
| 에러 처리 누락 보강 | AC 매핑 불가 (테스트 자체 부재) |
| 타입 힌트 보강 | 보안 취약점 (설계 수준) |
| 주석/docstring 보강 | 성능 문제 (알고리즘 변경 필요) |

**판단 기준:**
> "이 수정이 기존 테스트를 깨뜨리거나, 파일 구조를 바꾸는가?"
> → Yes: back 필요 (구조적 문제)
> → No: 직접 수정 (경미한 수정)

## 5. 리뷰 축 상세

### 5-1. 재사용성

- 동일/유사 로직이 여러 곳에 반복되지 않는가?
- 기존 유틸리티/헬퍼를 활용할 수 있는데 새로 작성하지 않았는가?
- 추출하면 다른 기능에서도 쓸 수 있는 공통 로직이 있는가?

**경미한 수정 예시**: 3줄 이상 반복 코드를 헬퍼로 추출
**back 예시**: 공통 모듈 신설이 필요한 수준의 중복 → `/workflow back design`

### 5-2. 코드 품질

- 함수/변수 네이밍이 의도를 명확히 전달하는가?
- 함수가 단일 책임을 가지는가? (너무 길거나 복잡하지 않은가?)
- 에러 처리가 적절한가? (silent fail, 빈 catch 등)
- 기존 코드베이스 패턴과 일관적인가?

**경미한 수정 예시**: `data` → `user_notifications`, 빈 except 블록에 로깅 추가
**back 예시**: 함수 분리가 파일 구조 변경을 요구 → `/workflow back design`

### 5-3. 런타임 효율

- N+1 쿼리가 있지 않은가?
- 불필요한 메모리 할당이 있지 않은가?
- 루프 내 불필요한 연산이 있지 않은가?
- Spec의 성능 요구사항(C-xxx)을 충족하는가?

**경미한 수정 예시**: `select_related` 추가, 리스트 컴프리헨션 최적화
**back 예시**: 알고리즘 자체 변경 필요 → `/workflow back design`

### 5-4. 보안

- 사용자 입력이 적절히 검증/이스케이프되는가?
- SQL injection, XSS, CSRF 취약점이 없는가?
- 인증/인가가 올바르게 적용되는가?
- 민감 정보가 로그/응답에 노출되지 않는가?

**경미한 수정 예시**: 입력 검증 추가, 로그에서 민감 필드 마스킹
**back 예시**: 인증 구조 자체 결함 → `/workflow back design`

### 5-5. AC 커버리지

Spec의 모든 AC를 코드/테스트에서 추적:

```
AC 커버리지 검증 결과:

| AC  | 슬라이스 | 테스트 | 상태 |
|-----|---------|--------|------|
| R-1 | A-1, A-2 | test_notification_created | ✅ 추적 가능 |
| R-2 | A-2 | test_signal_handler | ✅ 추적 가능 |
| R-3 | B-1 | test_list_view | ✅ 추적 가능 |
| C-1 | B-1 | test_rest_api_format | ✅ 추적 가능 |
| R-4 | B-2 | — | ❌ 테스트 누락 |
```

- ❌가 있으면 → 해당 슬라이스로 back (`/workflow back --slice N`)
- AC 자체가 부적절하면 → `/workflow back specify`

### 일반 진입

- 전체 코드 대상 리뷰

- **reworked 슬라이스 중심** 리뷰 (나머지는 이전 리뷰 유지)

## 7. 에스컬레이션 규칙

### 에스컬레이션 판단 기준

| 발견 사항 | 판단 | 에스컬레이션 대상 |
|-----------|------|-----------------|
| 테스트 누락 (AC 매핑 불가) | 슬라이스 문제 | `/workflow back --slice N` |
| 코드 품질 이슈 (경미) | 직접 수정 | Verify 내에서 수정 + 커밋 |
| 보안 취약점 (설계 수준) | 설계 문제 | `/workflow back design` |
| 성능 문제 (알고리즘 변경 필요) | 설계 문제 | `/workflow back design` |
| AC 자체가 부적절 | 요구사항 문제 | `/workflow back specify` |
| needs_rework ≥ 50% | 전체 설계 문제 | `/workflow back design` 권고 |

### Verify → Implement back 시 슬라이스 처리

- 특정 슬라이스만 back: `/workflow back --slice B-2 "에러 처리 누락"`
  - B-2만 `needs_rework`, 나머지 유지
- 전체 back: `/workflow back implement`
  - 모든 슬라이스 재검토 대상

## 8. Viewpoint 카탈로그

리뷰 축(§5)은 항상 적용된다. 추가로, 기능 특성에 따라 활성화되는 심화 viewpoint:

| Viewpoint | 적용 조건 | 주요 검토 내용 |
|-----------|-----------|-------------|
| 성능 심화 | DB 쿼리·대량 데이터·응답 시간 SLA 시 | 쿼리 플랜, 인덱스 활용, 캐시 전략 |
| 접근성 | UI 변경이 있는 기능 | WCAG, 스크린 리더, 키보드 네비게이션 |
| 비즈니스 로직 | 도메인 복잡도 높을 때 | 실제 비즈니스 의도와 구현 일치 |
| 동시성 | 멀티스레드·비동기·분산 처리 시 | 레이스 컨디션, 데드락, 순서 보장 |

---

## 9. 소요 시간

- 코드 리뷰이므로 **코드 양에 비례**
- Implement보다 짧음 (읽기 + 경미한 수정만)
- 부분 재진입 시: 해당 슬라이스 중심 리뷰로 더 빠름

## 10. Verify 체크리스트

### AI 코드 리뷰 (자동 실행)

- [ ] 재사용성 검토 완료 (중복 코드, 기존 유틸리티 활용)
- [ ] 코드 품질 검토 완료 (네이밍, 구조, 에러 처리)
- [ ] 런타임 효율 검토 완료 (N+1, 불필요한 할당)
- [ ] 보안 검토 완료 (입력 검증, 인증/인가, 민감 정보 노출)
- [ ] AC 커버리지 최종 확인 (모든 R-xxx/C-xxx 추적 가능)

### 경미한 수정

- [ ] 수정 후 전체 테스트 통과하는가?
- [ ] 수정이 파일 구조를 변경하지 않는가?
- [ ] 수정 커밋이 생성되었는가?

### 리뷰 완료 (승인 전)

- [ ] (기어 3) 인간이 리뷰 결과를 확인했는가?
- [ ] (기어 3) 비즈니스 로직이 실제 의도와 일치하는가?
- [ ] 에스컬레이션 필요 사항이 모두 처리되었는가?
- [ ] (기어 3) 인간이 명시적으로 "승인"했는가?
- [ ] (기어 2) AI 리뷰에서 구조적 이슈가 없는가? → 자동 승인

### 에스컬레이션 체크
